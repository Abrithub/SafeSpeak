# Requirements Document

## Introduction

SafeSpeak currently allows users to register with a username, password, and email address, but login only accepts a username and password. This creates a confusing experience — users who sign up with their email cannot use it to log in. This feature completes the email-based authentication flow by enabling login with email, enforcing email uniqueness at registration, and ensuring the existing password reset flow (which already uses email) remains consistent with the new login capability.

## Glossary

- **Auth_Service**: The backend Express router at `server/routes/auth.js` responsible for all authentication operations.
- **Login_Form**: The frontend React component at `client/frontend/src/pages/Login.jsx` that collects user credentials.
- **SignUp_Form**: The frontend React component at `client/frontend/src/pages/SignUp.jsx` that collects registration details.
- **API_Client**: The frontend module at `client/frontend/src/services/api.js` that communicates with the Auth_Service.
- **User**: A registered account stored in MongoDB with fields: `username`, `email`, `password`, `googleId`, `role`, `resetCode`, `resetCodeExpiry`.
- **Credential**: Either a username or an email address used to identify a User during login.
- **Reset_Code**: A 6-digit numeric code generated by the Auth_Service and sent to a User's email for password reset.
- **Mailer**: The module at `server/config/mailer.js` responsible for sending transactional emails via nodemailer.

## Requirements

### Requirement 1: Login with Email or Username

**User Story:** As a registered user, I want to log in using either my email address or my username, so that I don't have to remember which identifier I signed up with.

#### Acceptance Criteria

1. WHEN a login request is submitted with a valid email address and correct password, THE Auth_Service SHALL authenticate the User and return a signed JWT token.
2. WHEN a login request is submitted with a valid username and correct password, THE Auth_Service SHALL authenticate the User and return a signed JWT token.
3. WHEN a login request is submitted with an identifier that does not match any username or email in the database, THE Auth_Service SHALL return HTTP 401 with the message "Invalid credentials".
4. WHEN a login request is submitted with a correct identifier but an incorrect password, THE Auth_Service SHALL return HTTP 401 with the message "Invalid credentials".
5. THE Login_Form SHALL accept a single input field labeled "Email or Username" in place of the current username-only field.
6. WHEN the user submits the Login_Form, THE API_Client SHALL send the submitted value as an `identifier` field alongside `password` to the login endpoint.
7. IF the login response contains a token, THE Login_Form SHALL store the token in localStorage and redirect the user to the appropriate page based on their role.

---

### Requirement 2: Email Uniqueness Enforcement at Registration

**User Story:** As a new user, I want to be told immediately if my email is already registered, so that I can recover my existing account instead of creating a duplicate.

#### Acceptance Criteria

1. WHEN a registration request is submitted with an email address already associated with an existing User, THE Auth_Service SHALL return HTTP 409 with the message "Email already registered".
2. WHEN a registration request is submitted with a username already associated with an existing User, THE Auth_Service SHALL return HTTP 409 with the message "Username already taken".
3. THE Auth_Service SHALL treat email addresses as case-insensitive during uniqueness checks (e.g., "User@Example.com" and "user@example.com" are the same).
4. WHEN a registration request is submitted without an email address, THE Auth_Service SHALL return HTTP 400 with the message "Email is required".
5. WHEN a registration request is submitted with an email address that does not conform to a valid email format, THE Auth_Service SHALL return HTTP 400 with the message "Invalid email format".

---

### Requirement 3: Password Reset Flow Consistency

**User Story:** As a user who has forgotten my password, I want the reset flow to work reliably with the email I registered with, so that I can regain access to my account.

#### Acceptance Criteria

1. WHEN a forgot-password request is submitted with an email address that matches a registered User, THE Auth_Service SHALL generate a 6-digit Reset_Code, store it with a 15-minute expiry, and send it via the Mailer.
2. WHEN a forgot-password request is submitted with an email address that does not match any registered User, THE Auth_Service SHALL return HTTP 404 with the message "No account found with that email".
3. WHEN a reset-password request is submitted with a valid email, a matching Reset_Code, and a new password of at least 6 characters, THE Auth_Service SHALL update the User's password and clear the Reset_Code and its expiry.
4. WHEN a reset-password request is submitted with a Reset_Code that has passed its expiry time, THE Auth_Service SHALL return HTTP 400 with the message "Reset code has expired".
5. WHEN a reset-password request is submitted with a Reset_Code that does not match the stored code for the given email, THE Auth_Service SHALL return HTTP 400 with the message "Invalid reset code".
6. THE ForgotPassword page SHALL display a link to the Login_Form so users can navigate back after a successful reset.

---

### Requirement 4: Google OAuth Account Linking

**User Story:** As a user who previously signed in with Google, I want my Google account to be linked to my email so that I can also use email-based login if I set a password later.

#### Acceptance Criteria

1. WHEN a Google OAuth sign-in is completed and the Google account's email matches an existing User created via email registration, THE Auth_Service SHALL link the `googleId` to that existing User rather than creating a duplicate account.
2. WHEN a Google OAuth sign-in is completed and no existing User matches the Google account's email, THE Auth_Service SHALL create a new User with the Google email, `googleId`, and a generated username.
3. WHILE a User account has no password set (i.e., was created via Google OAuth only), THE Auth_Service SHALL reject any email/password login attempt for that account with HTTP 401 and the message "This account uses Google sign-in. Please continue with Google."

---

### Requirement 5: Frontend Login Form Update

**User Story:** As a user, I want the login page to clearly indicate that I can use either my email or username, so that I am not confused about which credential to enter.

#### Acceptance Criteria

1. THE Login_Form SHALL display a single text input with the placeholder "Email or Username".
2. THE Login_Form SHALL display a "Forgot password?" link that navigates to the ForgotPassword page.
3. WHEN the Login_Form is submitted with an empty identifier field, THE Login_Form SHALL display the validation message "Please enter your email or username" without making a network request.
4. WHEN the Auth_Service returns an error response to a login attempt, THE Login_Form SHALL display the error message returned by the Auth_Service.
5. THE Login_Form SHALL preserve the existing Google OAuth sign-in button and its behavior.
